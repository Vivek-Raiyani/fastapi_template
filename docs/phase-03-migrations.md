# Phase 3 — Migrations (Alembic)

Schema-as-code — restore any environment by running migrations instead of restoring DB dumps.

## Files

| File | Purpose |
|------|---------|
| `alembic.ini` | Alembic config entry point |
| `alembic/env.py` | Loads settings + models, runs async migrations |
| `alembic/script.py.mako` | Template for new migration files |
| `alembic/versions/` | Revision history |

## Django equivalent

| Django | Alembic |
|--------|---------|
| `python manage.py makemigrations` | `python manage.py makemigrations -m "message"` |
| `python manage.py migrate` | `python manage.py migrate` |
| `migrations/0001_initial.py` | `alembic/versions/001_initial_users_table.py` |

## Commands

```bash
# Apply all pending migrations
python manage.py migrate

# Apply up to a specific revision
python manage.py migrate -r 001

# Generate migration from model changes
python manage.py makemigrations -m "add profile table"

# Review generated file in alembic/versions/ before migrating!
python manage.py migrate
```

## Initial migration

`001_initial_users_table.py` creates the `users` table with all columns including `google_id` for OAuth.

## How it works

1. `alembic/env.py` imports `settings.DATABASE_URL` and `Base.metadata`
2. All models in `database/models/` are registered via imports
3. Autogenerate compares models ↔ database and writes a new revision
4. `upgrade()` applies SQL; `downgrade()` rolls back

## Fresh database workflow

```bash
# Delete old DB (SQLite)
del db.sqlite3

# Recreate from scratch
python manage.py migrate
python manage.py createsuperuser
```

For PostgreSQL: create empty database, then `migrate`.

## Best practices

- **Never** edit applied migrations in production — create a new revision
- Always review autogenerated migrations before applying
- Commit `alembic/versions/` to git — this is your schema history
- Use migrations, not `Base.metadata.create_all()`, in production
